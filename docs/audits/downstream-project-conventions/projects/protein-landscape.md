## 1. Project Center Of Gravity

- **Primary shape:** hybrid (research + software). The project is a research investigation with three first-class hypotheses and a heavy active question/interpretation pipeline, *and* it ships a non-trivial TypeScript/React+Three.js viewer (`viewer/`, 166 tracked files), a generated LinkML data-model package (`src/protein_landscape/model/`), and a Snakemake workflow (`workflow/`, 113 tracked files).
- **Dominant tracked directories:** `doc/` (310 files), `viewer/` (166), `workflow/` (113), `tests/` (49), `src/` (22), `descriptors/` (19), `docs/` (12, all under `docs/superpowers/`).
- **Dominant untracked / symlinked directories:** `data/` is a symlink to `/data/proj/protein-landscape` (3 entries); `results/` (11 entries, gitignored), `logs/` (19 entries, gitignored), `models/` (1 entry, gitignored), `.snakemake/`, `.venv/`.
- **`science.yaml` profile / aspects:** `profile: research`, `aspects: [computational-analysis, software-development]`.
- **Does profile + aspects capture the real project?** *partial*. The `software-development` aspect captures the viewer subproject, but the dual presence of (a) a Snakemake research pipeline, (b) a React/TS viewer subproject with its own `package.json`/husky/commitlint, and (c) a LinkML-generated cross-language data-model package is not really expressible as one `profile: research` + flat aspects list. The project effectively runs three concurrent build chains (uv/pytest, vite/vitest, snakemake) that the profile model does not enumerate. **Classification: candidate evidence for multi-axis project shape.**
- **Multi-axis needed?** *yes*. The viewer is a self-contained TypeScript subproject inside a Python research repo. A single profile can't cleanly express "research repo whose deliverable includes a viewer app and a generated cross-language data model"; the project has invented `aspects: [computational-analysis, software-development]` plus the LinkML-generated model package to cope. Classification: *recurring candidate convention* (multi-axis project) — see §10.

## 5. Task Lifecycle

- **`tasks/active.md` shape:** 1,112 lines, 101 task headings of the form `## [tNNN] title` followed by `- priority: ...`, `- status: ...`, `- aspects: [...]`, `- created: YYYY-MM-DD`, optional `- completed: YYYY-MM-DD`, then optional free-text body.
- **Done archive pattern:** `tasks/done/2026-04.md` (878 lines) — month-bucketed archive, same task shape.
- **Completed tasks remaining in active files:** *yes*. Of 101 entries in `tasks/active.md`, 44 carry `status: done`. Others: `proposed: 50, blocked: 4, deferred: 2, retired: 1`. No entries have a non-canonical Science task status. **The done backlog in active.md is large**: e.g. t001–t010 (background reading) all done since 2026-04-02 but never moved. *accidental drift* — archival convention exists but is incompletely enforced.
- **Task ID format:** `tNNN`, no zero-padding, monotonically increasing across active+done. Highest seen: `t167`. *recurring candidate convention*.
- **Status values observed (across active.md):** `proposed (50), done (44), blocked (4), deferred (2), retired (1)`. `proposed` is unusual — many other Science projects use `open` or `todo`. *valid project-specific convention* if intentional; otherwise mild drift.
- **Use of `priority` / `aspects` / `group` / `related` / `blocked-by` / `created` / `completed`:** `priority` and `aspects` are uniformly used; `created` always present; `completed` only on done tasks. **`group` and `blocked-by` not observed** in `tasks/active.md`. **`related` not present in tasks** — relations live one-way: questions/hypotheses/plans link to `task:tNNN`, not the reverse. *missing back-references*.
- **Code, data, reports, interpretations link back to task IDs:** *partial*. Plans (`doc/plans/*`) and interpretations (`doc/interpretations/*`) routinely reference `task:tNNN`. Code does not — `workflow/scripts/*.py` files do not carry task-id headers; only the descriptor JSONs encode `git_commit` (not `task`). `next-steps-2026-04-24.md` flags this drift explicitly: "Viewer refactor work started without being tracked. Four new plan docs ... exist with active viewer code in flight ..., but no task in `tasks/active.md` anchors this slice."
- **Are task groups a stable concept worth modeling:** `aspects` does some grouping (`computational-analysis` vs `software-development`). No `group` or `epic` field. The de-facto grouping is by `tNNN`-prefix patterns (`t14[3-5]` for the q63 step series, `t15[1-3]` for the cross-lens-Moran-I/TED-domain/SAE batch). Not a strong signal for a `group` field yet. *project-specific convention* (de-facto only).
- **Do tasks contain enough provenance, or should they link to result entities:** *insufficient*. Done tasks have a free-text body with paths/findings (e.g. t034 mentions `workflow/scripts/map_taxonomy.py`), but no structured `produces:` or `result:` link to interpretation/descriptor entities. The forward chain is: `task → interpretation → finding (inline) → hypothesis prediction`. ## 9. Project-Grown Conventions

- **Frictionless-style descriptor sidecars** — `descriptors/<name>.parquet.descriptor.json` (19 files). Carries `artifact, produced_at, git_commit, tool_version, inputs[{path,sha256}], outputs[{path,sha256,row_count}], command, parameters, notes`. *useful for ≥2 projects* (most computational-analysis projects produce parquets with similar provenance needs); *likely missing upstream support*; *worth a future focused design pass* — this is essentially "datapackage-lite, output-anchored, sha256-sealed".
- **LinkML-driven cross-language data model** — `src/protein_landscape/model/{schema.yaml, codegen.py, generated/, semantic_checks.py}` + `knowledge/ontology/pl.ttl` + a pre-commit hook regenerating artifacts. Targets Pydantic, TypeScript, JSON Schema, TTL, parquet contracts. Coupled to the viewer subproject (TS targets land in `viewer/`). *project-specific* in detail, but the *concept* (codegen-from-schema for a project's entity model) is a likely recurring need; *worth a future design pass* once at least one other downstream project shows the same shape.
- **Per-hypothesis synthesis rollup directory** — `doc/reports/synthesis/{_emergent-threads.md, h0[1-3]-*.md}`. Generated from hypothesis specs + interpretation refs + the knowledge graph; carries `source_commit`, `provenance_coverage`, and a graph-derived orphan accounting (`orphan_question_count`, `orphan_interpretation_count`, `orphan_ids: [...]`). *useful for ≥2 projects*; *likely missing upstream support* — most large projects with hypotheses would benefit from a comparable rollup.
- **Inline `finding:F[0-9]+` numbered-claim convention** — 17 distinct findings cited across the corpus with no backing entity files. *project-grown* alternative to a `proposition:` / `claim:` entity family. *worth a future design pass* — choose between an inline-claim convention (lightweight) and explicit entities (more graph-friendly).
- **`access:` structured block on dataset frontmatter** — see `doc/datasets/deeploc2.md`. Encodes `level, verified, verification_method, last_reviewed, verified_by, source_url, credentials_required, exception: {mode, decision_date, followup_task, superseded_by_dataset, rationale}`. Captures the data-acquisition lifecycle as metadata. *useful for ≥2 projects*; *likely missing upstream support*.
- **Top-level `descriptors/` directory** — 19 sidecars, all parquet outputs. Distinct from `data/` (payloads, gitignored) and `results/` (run-scoped, gitignored). *project-grown*; the placement (top-level rather than under `data/` or `results/`) is unusual.
- **Chained `prior_analyses` in next-steps** — `doc/meta/next-steps-YYYY-MM-DD.md` files form a temporal chain via `prior_analyses: [meta:next-steps-YYYY-MM-DD]`. *project-specific*; *interesting observation* but not obviously upstream-relevant.
- **`runpod/` cloud-execution scripts** — 8 shell scripts for A100/H100 GPU runs (`push_to_runpod.sh`, `pull_results.sh`, `run.sh`, etc.) with a README capturing instance specs and disk requirements. *useful for ≥2 projects* (any project with GPU pipelines); *likely missing upstream support*. Compare with peer projects' execution-environment notes.
- **`doc/issues/` refactor staging** — 4 files in pairs (`*-desired_file_structure.md` + `*-files_to_remove.md`). Matches the user-global CLAUDE.md "Refactoring" instructions ("Prior to refactoring, document desired file structure and clean-up steps"). *project-grown* but driven by a global user convention.
- **`docs/superpowers/{plans,specs}/`** — agent-authored, very-long-form (568–3,421 lines) implementation plans + design specs, kept separate from human-curated `doc/plans/` and `doc/specs/`. *useful for ≥2 projects* if other projects also use Superpowers; *likely missing upstream support* (Science's `doc/` model doesn't cleanly partition human vs agent authoring).
- **`papers/references.bib` + `cite:Year` namespace** — top-level BibTeX file backing all `cite:Author2025` references in entity frontmatter. *recurring candidate convention*; this is how research projects naturally connect to a literature graph.
- **`doc/figures/dags/` with `<slug>.dot` + `<slug>.edges.yaml`** — DAG figures kept in DOT plus a structured edges.yaml carrying `relation, edge_status, identification, description, data_support: [{task, description}], lit_support: [{paper, description}]`. *project-grown* causal-DAG companion data. *worth a future design pass* — this is more structured than a typical plot.
- **`spec:` two-file convention at the project root** — `specs/research-question.md` + `specs/scope-boundaries.md` as canonical project-wide statements. *recurring candidate convention*; cross-check with peer projects.
